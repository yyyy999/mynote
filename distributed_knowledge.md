# 分布式计算与GPU架构知识科普

本文档旨在帮助理解 TileScale 项目中涉及的分布式计算概念和底层架构知识。

---

## 一、GPU 架构基础

### 1.1 GPU 的层次结构

现代 GPU (如 NVIDIA H100) 采用层次化的计算架构：

```
┌─────────────────────────────────────────────────────────────┐
│                        GPU (Device)                          │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │                      GPC (GPU处理簇)                     │ │
│  │  ┌─────────────────────────────────────────────────────┐│ │
│  │  │                    SM (流多处理器)                   ││ │
│  │  │  ┌─────────────────────────────────────────────────┐││ │
│  │  │  │              Warp (线程束, 32线程)              │││ │
│  │  │  │  ┌─────────────────────────────────────────────┐│││ │
│  │  │  │  │            Thread (线程)                    ││││ │
│  │  │  │  └─────────────────────────────────────────────┘│││ │
│  │  │  └─────────────────────────────────────────────────┘││ │
│  │  └─────────────────────────────────────────────────────┘│ │
│  └─────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

**各层次说明**：

| 层次 | 数量(H100) | 说明 |
|------|-----------|------|
| SM (Streaming Multiprocessor) | 132个 | 独立的计算单元，可执行一个线程块 |
| Warp | 每SM多个 | 32个线程为一组，同步执行 |
| Thread | 每Warp 32个 | 最小执行单元 |
| Tensor Core | 每SM多个 | 专门用于矩阵乘法的硬件单元 |

### 1.2 GPU 内存层次

```
┌─────────────────────────────────────────────────────────────┐
│                    HBM (高带宽内存)                          │
│                    容量: 80GB (H100)                         │
│                    带宽: 3.35TB/s                            │
├─────────────────────────────────────────────────────────────┤
│                    L2 Cache                                  │
│                    容量: 50MB (H100)                         │
│                    延迟: ~100 cycles                         │
├─────────────────────────────────────────────────────────────┤
│              Shared Memory / L1 Cache (每SM)                 │
│              容量: 228KB (H100)                              │
│              延迟: ~30 cycles                                │
├─────────────────────────────────────────────────────────────┤
│              Registers (每线程)                              │
│              容量: 256KB/SM                                  │
│              延迟: ~1 cycle                                  │
└─────────────────────────────────────────────────────────────┘
```

**关键概念**：
- **延迟**: 从请求数据到获得数据的时间
- **带宽**: 单位时间内传输的数据量
- **越靠近计算单元的内存，延迟越低，容量越小**

## 二、多GPU通信机制

### 2.1 为什么需要多GPU通信？

单个 GPU 的内存和算力有限，无法满足大模型训练需求：

```
GPT-4 估计参数量: ~1.8万亿参数
FP16 存储: 3.6TB
单张 H100 内存: 80GB
需要 GPU 数量: ~45张 (仅存储参数)
```

### 2.2 GPU 互连技术

#### 2.2.1 NVLink

NVIDIA 专有的高速互连技术：

```
┌──────────┐                    ┌──────────┐
│  GPU 0   │ ═══════════════════│  GPU 1   │
│          │     NVLink         │          │
│ HBM 80GB │   900GB/s双向      │ HBM 80GB │
└──────────┘                    └──────────┘
```

**特点**：
- 带宽: 900GB/s (H100 NVLink 4.0)
- 延迟: ~5μs
- 支持直接内存访问 (GPU可直接读写另一GPU的内存)

#### 2.2.2 PCIe

标准的外设互连：

```
┌──────────┐                    ┌──────────┐
│  GPU 0   │ ──────────────────│  GPU 1   │
│          │      PCIe         │          │
│          │    64GB/s双向      │          │
└──────────┘                    └──────────┘
```

**特点**：
- 带宽: 64GB/s (PCIe 5.0 x16)
- 延迟: ~10μs
- 通用性强，但带宽较低

#### 2.2.3 InfiniBand

节点间高速网络：

```
┌─────────────────┐                    ┌─────────────────┐
│    Node 0       │                    │    Node 1       │
│  ┌─────┐ ┌─────┐│                    │┌─────┐ ┌─────┐  │
│  │GPU 0│ │GPU 1││                    ││GPU 0│ │GPU 1│  │
│  └──┬──┘ └──┬──┘│                    │└──┬──┘ └──┬──┘  │
│     └────┬────┘ │                    │ ┌─┴────┘       │
│          │      │                    │ │              │
│     ┌────┴────┐ │   InfiniBand       │ │  ┌────────┐  │
│     │  NIC    │═│════════════════════│═│═│   NIC  │  │
│     │400Gb/s  │ │   NDR 400Gb/s      │ │  │400Gb/s │  │
│     └─────────┘ │                    │ │  └────────┘  │
└─────────────────┘                    └─────────────────┘
```

**特点**：
- 带宽: 400Gb/s (NDR InfiniBand)
- 延迟: ~1μs
- 支持远程直接内存访问 (RDMA)

### 2.3 GPU 通信模式

#### 2.3.1 点对点通信 (P2P)

```
GPU 0                          GPU 1
┌──────────┐                   ┌──────────┐
│  数据A   │ ──── Put ────────>│  数据A'  │
│          │<──── Get ──────── │          │
└──────────┘                   └──────────┘

Put: 主动发送数据到目标GPU
Get: 主动从目标GPU拉取数据
```

#### 2.3.2 CUDA IPC (进程间通信)

允许不同进程共享 GPU 内存：

```python
# 进程 0
tensor = torch.randn(1000, device='cuda')
handle = cudaIpcGetMemHandle(tensor.data_ptr())  # 获取IPC句柄
# 将 handle 发送给进程 1

# 进程 1
# 接收 handle
ptr = cudaIpcOpenMemHandle(handle)  # 打开IPC句柄，获得指针
tensor = torch.from_ptr(ptr, shape=(1000,))  # 从指针创建张量
```

## 三、NVSHMEM 详解

### 3.1 什么是 NVSHMEM？

**NVSHMEM** (NVIDIA SHMEM) 是 NVIDIA 提供的分布式共享内存库，它：

1. **抽象化远程内存**: 让多 GPU 看起来像一个共享内存系统
2. **提供对称堆**: 所有 GPU 分配相同大小的内存区域
3. **支持 RDMA**: 直接远程内存访问，无需 CPU 参与

### 3.2 对称堆 (Symmetric Heap)

```
┌─────────────────────────────────────────────────────────────┐
│                      对称堆布局                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  GPU 0              GPU 1              GPU 2              GPU 3
│  ┌─────────┐       ┌─────────┐       ┌─────────┐       ┌─────────┐
│  │ addr 0  │       │ addr 0  │       │ addr 0  │       │ addr 0  │
│  │         │       │         │       │         │       │         │
│  │ 数据A   │       │ 数据B   │       │ 数据C   │       │ 数据D   │
│  │         │       │         │       │         │       │         │
│  │ addr N  │       │ addr N  │       │ addr N  │       │ addr N  │
│  └─────────┘       └─────────┘       └─────────┘       └─────────┘
│                                                             │
│  相同的虚拟地址空间，每个GPU存储自己的数据分片                  │
└─────────────────────────────────────────────────────────────┘
```

**关键特性**：
- 所有 GPU 上相同变量具有相同的虚拟地址
- GPU 0 可以直接访问 GPU 1 的内存：`data[1][i]` 
- 实现了 **PGAS** (Partitioned Global Address Space) 编程模型

### 3.3 NVSHMEM 通信 API

```c
// Put: 从本地发送到远程
nvshmem_putmem_nbi(dst, src, size, pe);  // 非阻塞
nvshmem_putmem(dst, src, size, pe);       // 阻塞

// Get: 从远程拉取到本地
nvshmem_getmem_nbi(dst, src, size, pe);  // 非阻塞
nvshmem_getmem(dst, src, size, pe);       // 阻塞

// Signal: 发送信号通知
nvshmem_put_signal_nbi(dst, src, size, sig_addr, signal, sig_op, pe);

// 等待信号
nvshmem_wait_until(sig_addr, cmp, value);

// 屏障同步
nvshmem_barrier_all();  // 所有PE同步
```

### 3.4 PE (Processing Element) 概念

在 NVSHMEM 中，每个 GPU 被称为一个 **PE** (Processing Element)：

```python
pe_id = nvshmem_my_pe()      # 获取当前 PE ID (0, 1, 2, ...)
num_pes = nvshmem_n_pes()    # 获取 PE 总数
```

## 四、集合通信操作

### 4.1 什么是集合通信？

集合通信是指**多个进程/GPU共同参与**的通信操作，常见的有：

| 操作 | 说明 | 数据流 |
|------|------|--------|
| Broadcast | 一对多广播 | 1 → N |
| Reduce | 多对一规约 | N → 1 |
| AllReduce | 全局规约 | N ↔ N |
| AllGather | 全局收集 | N ↔ N |
| All2All | 全交换 | N ↔ N |
| ReduceScatter | 规约+分散 | N ↔ N |

### 4.2 AllGather 详解

**功能**: 每个 GPU 贡献自己的数据，最终所有 GPU 都拥有完整数据

```
初始状态:
GPU 0: [A]     GPU 1: [B]     GPU 2: [C]     GPU 3: [D]

AllGather 后:
GPU 0: [A,B,C,D]  GPU 1: [A,B,C,D]  GPU 2: [A,B,C,D]  GPU 3: [A,B,C,D]
```

#### Ring AllGather 算法

将 GPU 排成环形，分步传递数据：

```
Step 1: 每个GPU发送自己的数据给右邻居
        GPU 0 ──[A]──> GPU 1 ──[B]──> GPU 2 ──[C]──> GPU 3 ──[D]──> GPU 0
        结果: GPU 0 有 [D], GPU 1 有 [A], GPU 2 有 [B], GPU 3 有 [C]

Step 2: 发送上一步收到的数据
        GPU 0 ──[D]──> GPU 1 ──[A]──> GPU 2 ──[B]──> GPU 3 ──[C]──> GPU 0
        结果: GPU 0 有 [C,D], GPU 1 有 [D,A], GPU 2 有 [A,B], GPU 3 有 [B,C]

Step 3: 继续...
Step 4: 所有GPU都有完整数据 [A,B,C,D]
```

**时间复杂度**: O(N-1) 步，每步传输数据量 = 总数据量/N

**优点**: 带宽利用率高，适合大数据量

### 4.3 All2All 详解

**功能**: 每个 GPU 向每个其他 GPU 发送不同的数据

```
初始状态 (每个GPU有4块数据，分别要发给4个GPU):
GPU 0: [A0, A1, A2, A3]   # A0给GPU0, A1给GPU1, ...
GPU 1: [B0, B1, B2, B3]
GPU 2: [C0, C1, C2, C3]
GPU 3: [D0, D1, D2, D3]

All2All 后:
GPU 0: [A0, B0, C0, D0]   # 收集所有GPU发给自己的第0块
GPU 1: [A1, B1, C1, D1]
GPU 2: [A2, B2, C2, D2]
GPU 3: [A3, B3, C3, D3]
```

**应用场景**: MoE 专家路由、数据重分布

### 4.4 AllReduce 详解

**功能**: 所有 GPU 的数据进行规约操作（求和、最大值等），结果广播给所有 GPU

```
初始状态:
GPU 0: [1, 2, 3]     GPU 1: [4, 5, 6]     GPU 2: [7, 8, 9]

AllReduce (Sum) 后:
GPU 0: [12, 15, 18]  GPU 1: [12, 15, 18]  GPU 2: [12, 15, 18]
       (1+4+7)       (2+5+8)              (3+6+9)
```

**应用场景**: 分布式训练中的梯度同步

### 4.5 ReduceScatter 详解

**功能**: 规约 + 分散，每个 GPU 只获得规约结果的一部分

```
初始状态:
GPU 0: [1, 2, 3]     GPU 1: [4, 5, 6]     GPU 2: [7, 8, 9]

ReduceScatter (Sum) 后:
GPU 0: [12]          GPU 1: [15]          GPU 2: [18]
       (1+4+7)              (2+5+8)              (3+6+9)
```

**应用场景**: 张量并行中的部分规约

## 五、MoE (Mixture of Experts) 专家路由

### 5.1 什么是 MoE？

MoE 是一种模型架构，将大模型拆分为多个"专家"网络：

```
传统模型:
┌─────────────────────────────────────┐
│           单一大模型                 │
│         (所有token经过相同计算)       │
└─────────────────────────────────────┘

MoE 模型:
┌─────────────────────────────────────┐
│              Router                 │
│         (决定token去哪个专家)         │
└──────┬──────┬──────┬──────┬────────┘
       │      │      │      │
   ┌───┴──┐ ┌─┴───┐ ┌┴────┐ ┌┴────┐
   │Expert│ │Expert│ │Expert│ │Expert│
   │  0   │ │  1   │ │  2   │ │  3   │
   └──────┘ └──────┘ └──────┘ └──────┘
```

### 5.2 MoE 中的 Token 路由

```
输入 Token: [T0, T1, T2, T3, T4, T5, T6, T7]

Router 决定:
T0 → Expert 1
T1 → Expert 3
T2 → Expert 0
T3 → Expert 2
T4 → Expert 1
T5 → Expert 0
T6 → Expert 3
T7 → Expert 2

重排后发给各专家:
Expert 0: [T2, T5]
Expert 1: [T0, T4]
Expert 2: [T3, T7]
Expert 3: [T1, T6]
```

### 5.3 分布式 MoE 的通信需求

当专家分布在不同 GPU 上时：

```
┌─────────────────────────────────────────────────────────────┐
│                     MoE 分布式推理                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  GPU 0              GPU 1              GPU 2              GPU 3
│  ┌─────────┐       ┌─────────┐       ┌─────────┐       ┌─────────┐
│  │ Expert 0│       │ Expert 1│       │ Expert 2│       │ Expert 3│
│  │ [T2,T5] │       │ [T0,T4] │       │ [T3,T7] │       │ [T1,T6] │
│  └────┬────┘       └────┬────┘       └────┬────┘       └────┬────┘
│       │                 │                 │                 │
│       └─────────────────┴─────────────────┴─────────────────┘
│                         All2All 通信
│       ┌─────────────────┬─────────────────┬─────────────────┐
│       │                 │                 │                 │
│  ┌────┴────┐       ┌────┴────┐       ┌────┴────┐       ┌────┴────┐
│  │ 输出 T2 │       │ 输出 T0 │       │ 输出 T3 │       │ 输出 T1 │
│  │ 输出 T5 │       │ 输出 T4 │       │ 输出 T7 │       │ 输出 T6 │
│  └─────────┘       └─────────┘       └─────────┘       └─────────┘
│                                                             │
│  需要 All2All 将 token 发送到对应专家所在的 GPU               │
└─────────────────────────────────────────────────────────────┘
```

**通信流程**：
1. **Dispatch (分发)**: All2All 将 token 发送到专家所在 GPU
2. **Compute (计算)**: 各 GPU 执行专家计算
3. **Combine (合并)**: All2All 将结果返回原 GPU

## 六、分布式矩阵乘算法

### 6.1 问题背景

超大矩阵乘法无法在单 GPU 上完成：

```
矩阵维度: 16384 × 16384
FP16 存储: 512MB (单矩阵)
计算量: 16384³ ≈ 4.4 × 10¹² FLOPs
```

### 6.2 2D Mesh 拓扑

将多个 GPU 排列成 2D 网格：

```
4×4 Mesh (16个GPU):

        PE 0   PE 1   PE 2   PE 3
        ┌─────┬─────┬─────┬─────┐
   PE 0 │     │     │     │     │
        ├─────┼─────┼─────┼─────┤
   PE 1 │     │     │     │     │
        ├─────┼─────┼─────┼─────┤
   PE 2 │     │     │     │     │
        ├─────┼─────┼─────┼─────┤
   PE 3 │     │     │     │     │
        └─────┴─────┴─────┴─────┘

PE(pe_mn, pe_k) = pe_mn * MESH + pe_k
- pe_mn: 行/列索引
- pe_k: K维度索引
```

### 6.3 SUMMA 算法

**SUMMA** (Scalable Universal Matrix Multiplication Algorithm)

#### 矩阵分块

```
C = A × B

A 分块:          B 分块:          C 分块:
┌────┬────┐      ┌────┬────┐      ┌────┬────┐
│A00 │A01 │      │B00 │B01 │      │C00 │C01 │
├────┼────┤  ×   ├────┼────┤  =   ├────┼────┤
│A10 │A11 │      │B10 │B11 │      │C10 │C11 │
└────┴────┘      └────┴────┘      └────┴────┘

Cij = Σ Aik × Bkj
```

#### 算法流程

```
2×2 Mesh 示例:

初始分布:
PE(0,0): A00, B00    PE(0,1): A01, B01
PE(1,0): A10, B10    PE(1,1): A11, B11

迭代 k=0:
1. PE(*,0) 广播 A*0 到同行 PE
   PE(0,0) → PE(0,1): A00
   PE(1,0) → PE(1,1): A10

2. PE(0,*) 广播 B0* 到同列 PE
   PE(0,0) → PE(1,0): B00
   PE(0,1) → PE(1,1): B01

3. 本地计算:
   PE(0,0): C00 += A00 × B00
   PE(0,1): C01 += A00 × B01
   PE(1,0): C10 += A10 × B00
   PE(1,1): C11 += A10 × B01

迭代 k=1:
类似地广播 A*1 和 B1*，累加到 C
```

**通信量**: O(N²/√P) 其中 P 是 GPU 数量

### 6.4 Cannon 算法

**Cannon 算法** 通过环形移位实现矩阵乘

#### 初始对齐

```
初始分布后，先进行移位对齐:

A 矩阵移位 (每行向左移 pe_k 次):
┌────┬────┐      ┌────┬────┐
│A00 │A01 │  →   │A00 │A01 │  (第0行不移位)
├────┼────┤      ├────┼────┤
│A10 │A11 │  →   │A11 │A10 │  (第1行左移1次)
└────┴────┘      └────┴────┘

B 矩阵移位 (每列向上移 pe_mn 次):
┌────┬────┐      ┌────┬────┐
│B00 │B01 │  →   │B00 │B11 │  (第0列不移位, 第1列上移1次)
├────┼────┤      ├────┼────┤
│B10 │B11 │  →   │B10 │B01 │
└────┴────┘      └────┴────┘
```

#### 迭代计算

```
每次迭代:
1. 本地计算: C += A × B
2. A 向左移位一次
3. B 向上移位一次

迭代 √P 次后完成
```

**通信量**: O(N²/√P)，与 SUMMA 相同，但通信模式更规则

## 七、同步机制详解

### 7.1 为什么需要同步？

分布式计算中，各 GPU 执行速度不同，需要协调：

```
GPU 0: [计算完成] ────────────────────── [继续]
GPU 1: [计算中...] ──── [完成] ──────── [继续]
GPU 2: [计算完成] ────────────────────── [继续]
GPU 3: [计算中.........] ── [完成] ──── [继续]
                              ↑
                          需要等待所有GPU
```

### 7.2 Barrier (屏障)

```
Barrier: 所有 GPU 到达屏障后才能继续

GPU 0: ────┐
GPU 1: ────────┐
GPU 2: ──────┐ │
GPU 3: ────────────┐
                  │
          ┌───────┴───────┐
          │    Barrier    │
          └───────┬───────┘
                  │
GPU 0: ───────────┼───>
GPU 1: ───────────┼───>
GPU 2: ───────────┼───>
GPU 3: ───────────┼───>
```

### 7.3 Signal (信号) 同步

更细粒度的同步机制：

```
GPU 0 (发送方):                    GPU 1 (接收方):
┌─────────────────┐               ┌─────────────────┐
│ 1. 发送数据     │               │                 │
│ 2. 发送信号     │ ────────────> │ 3. 等待信号     │
│    signal=1     │               │    wait(signal) │
│                 │               │ 4. 接收数据     │
└─────────────────┘               └─────────────────┘
```

**Signal 操作类型**:
- `SIGNAL_SET`: 设置信号值
- `SIGNAL_ADD`: 原子加
- `SIGNAL_AND`: 原子与
- `SIGNAL_OR`: 原子或

### 7.4 Fence (栅栏)

确保内存操作顺序：

```
GPU 0:
┌─────────────────────────────────────┐
│ 写操作1                             │
│ 写操作2                             │
│ T.fence()  ← 确保之前的写操作完成    │
│ 写操作3                             │
└─────────────────────────────────────┘

其他 GPU 可以安全读取操作1、2的结果
```

## 八、TileScale 中的编程抽象

### 8.1 T.Scale 原语

TileScale 提供层次化的并行抽象：

```python
# 设备级并行
with T.Scale("device") as dev_id, dev_num:
    # dev_id: 当前设备ID (0, 1, 2, ...)
    # dev_num: 设备总数
    pass

# Block级并行
with T.Scale("block"):
    pass

# Warp级并行
with T.Scale("warp") as warp_id, warp_num:
    pass
```

### 8.2 分布式内存视图

```python
# 数据分布策略
A_global = T.view(A, layout=T.FullCol)   # 列分片
B_global = T.view(B, layout=T.FullRow)   # 行分片
C_global = T.view(C, layout=T.Replica)   # 副本(每个GPU都有完整数据)
```

## 九、性能优化要点

### 9.1 通信计算重叠

```
无重叠:
[通信] [计算] [通信] [计算]
─────────────────────────> 时间

有重叠:
[通信]
   [计算] [通信]
      [计算] [通信]
         [计算]
─────────────────────────> 时间
```

### 9.2 流水线化

```python
# 使用双缓冲实现流水线
for k in T.serial(K):
    # 缓冲区交替使用
    T.copy(A[k % 2], A_shared)  # 读取
    T.gemm(A[(k+1) % 2], B, C)  # 计算 (使用上一轮的数据)
```

### 9.3 数据局部性

```
优先使用更近的内存:
Register > Shared Memory > L2 Cache > HBM > NVLink > PCIe > InfiniBand
```
